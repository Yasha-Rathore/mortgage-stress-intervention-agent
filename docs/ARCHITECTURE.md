# Architecture

Detailed component design and design decisions.

---

## System overview

The system is a **multi-agent autonomous pipeline** triggered weekly by a
GitHub Actions cron. It reads customer state from Supabase, runs each
at-risk customer through a chain of LLM-based agents, executes channel-
specific actions, summarises responses, and produces an executive report —
all without human involvement until escalation.

Each component is independently testable and replaceable.

---

## Components

### 1. Orchestrator (`03_orchestrator.ipynb` / `scheduler.py`)

The top-level loop. Reads moderate-and-severe customers from Supabase,
calls the Triage Agent for each, dispatches to the correct sub-agent,
writes the result to the `interventions` table, and finally calls the
Reporter Agent to produce a weekly memo.

**Design choice:** the orchestrator does not make any LLM calls itself.
All reasoning is delegated to specialised agents so the loop logic remains
deterministic and debuggable.

---

### 2. Triage Agent

**Inputs:** customer profile (income, loan, savings, credit, etc) + last
4 weeks of behavioural signals.

**Outputs (strict JSON):**
- `stress_assessment` — low / moderate / high / critical
- `key_signals` — 2–3 signals driving the decision
- `reasoning` — 2-sentence natural-language explanation
- `action` — MONITOR / EMAIL / CALL / ESCALATE
- `action_rationale` — why this action over another
- `urgency` — routine / soon / immediate

**Model:** Groq Llama 3.3 70B with `temperature=0.1` for consistent decisions.

**Why an LLM here, not rule-based?** Customers showing identical numerical
stress signals can mean very different things in context (e.g. high CC
utilisation right after Christmas vs in May). The LLM weights signals
relative to each other in ways static rules cannot, and it produces
human-readable reasoning that is auditable.

---

### 3. Email Sub-Agent

When Triage returns `action: "EMAIL"`:

1. Calls Groq again with a separate prompt to draft a personalised email
   (subject + 3-paragraph body, warm and non-corporate).
2. Sends via SendGrid to the customer's email address (`YOUR_EMAIL`
   during dev/demo).
3. Marks `email_sent = true` in the customers table.

**Design choice:** the email prompt explicitly forbids mentioning risk,
stress, or arrears. This mirrors how banks word real customer-assist
outreach to avoid stigmatising the customer.

---

### 4. Voice Sub-Agent

When Triage returns `action: "CALL"`:

1. Calls Vapi.ai with the customer's phone number, a custom system
   prompt for "Maya", and the Groq model id.
2. Vapi handles the actual outbound call, voice synthesis (PlayHT), and
   real-time conversation.
3. When the call ends, Vapi POSTs the transcript to the Railway webhook
   (`/vapi-webhook`).
4. The webhook handler immediately calls the Summariser Agent on the
   transcript and updates the intervention row.

**Vapi free-tier limitation:** outbound calls are US-only. For the demo,
voice interventions are populated using Groq-generated mock transcripts
that are summarised through the real Summariser — so the dashboard shows
realistic voice-intervention data without burning credits or limitations
of the free tier.

---

### 5. Summariser Agent

**Inputs:** transcript (call) or email+reply, plus customer context.

**Outputs (strict JSON):**
- `sentiment` — positive / neutral / concerned / distressed
- `engaged` — true/false (did the customer respond meaningfully?)
- `reasons_disclosed` — list of reasons the customer mentioned (job loss,
  medical expenses, etc)
- `offer_accepted` — none / repayment_pause / interest_only / hardship_referral / booked_call
- `outcome` — resolved / monitor / follow_up / escalate_to_human
- `new_risk_signals` — anything new learned from the conversation
- `next_action` — what should happen in the next triage cycle
- `rm_briefing` — paragraph for human RM if escalating

**Why this is the most valuable agent:** structured intelligence from
conversations is what banks currently *cannot* get at scale. Manual
post-call coding is the bottleneck. This agent removes it.

---

### 6. Reporter Agent

Runs once at the end of each orchestration run. Reads all interventions
for the current week and writes a 3-paragraph executive memo:

1. Portfolio stress summary and channel mix
2. Intervention effectiveness and notable outcomes
3. Recommended focus areas for next week

**Design choice:** the report is generated by an LLM but the data points
fed to it are computed deterministically from Supabase. This avoids the
LLM hallucinating numbers.

## Data model

```
customers (500 rows)
  customer_id PK
  name, email, state, phone
  annual_income, loan_balance, monthly_repayment
  savings_balance, credit_card_balance, credit_card_limit
  stress_ratio, risk_tier
  intervention_status, email_sent, email_opened, email_replied

weekly_signals (4000 rows = 500 × 8 weeks)
  customer_id FK
  week, savings_balance, cc_utilisation
  days_repayment_late, irregular_income

interventions (~60-80 rows per run)
  customer_id FK
  week, action, channel
  triage_reasoning
  email_subject, email_body
  sentiment, offer_accepted, outcome
  rm_briefing, new_risk_signals

weekly_reports (1 row per week)
  week PK
  total_interventions, resolved, escalated
  email_open_rate, acceptance_rate
  report_text
```

---

## Design tradeoffs

**Single LLM provider (Groq).** Using one model for all five agents means
consistent latency and zero cost, but introduces a single point of failure.
In production a fallback to OpenAI or Anthropic would be wired up.

**Synchronous orchestration.** The loop processes customers serially. For
500 real customers this would take ~30 minutes. In production this would
move to a queue-based worker pattern (e.g. Cloud Tasks).

**LLM-generated email content.** Risk: hallucinated promises (e.g. "we
can refund your last 3 repayments") would be a compliance problem. The
prompt mitigates by explicitly forbidding promises and ending with two
fixed CTAs ("Book a call" / "I'm fine no thanks"). In production this
would be paired with a content filter and human review of templates.

**No retraining loop.** The agents do not learn from outcome data over
time. This is intentional for the prototype — production would feed
intervention outcomes back into the Triage prompt as few-shot examples.

---

## Security and compliance considerations

- All keys are environment variables, never committed to GitHub.
- The Supabase anon key has read-only access on the dashboard frontend.
- SendGrid sender identity is verified to prevent spoofing.
- Vapi calls use a verified bank caller-ID in production (US trial number
  in demo).
- Customer PII is synthetic — no real data is used.
- Row Level Security on Supabase is permissive for the demo; in production
  every read would require an authenticated `credit_risk_analyst` role.

---

## What would change in production

1. Real customer data feeds replacing synthetic data
2. RLS policies on every Supabase table, role-based access control
3. End-to-end audit logging of every agent decision for regulatory review
4. Human-in-the-loop approval for first 90 days of CALL actions
5. A/B testing framework comparing autonomous vs human-led interventions
6. Multi-LLM fallback (Groq primary, OpenAI secondary)
7. Twilio/Telnyx or carrier-grade voice provider with bank caller-ID
8. Compliance content filter on all customer-facing LLM output
9. Real-time analytics on intervention conversion by cohort
10. Integration with the bank's existing CRM and CDP
